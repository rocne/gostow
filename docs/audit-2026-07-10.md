# gostow audit — 2026-07-10

Scope: full read of the engine, CLI, getopt, ui and conformance harness against
GNU Stow 2.4.1's actual source (`.oracle/share/perl5/5.40/Stow.pm`, `Stow/Util.pm`,
`.oracle/bin/stow`) and against the running binary. Every finding marked
**confirmed** was reproduced in this session by running real stow 2.4.1 and gostow
side by side; the exact recipe is given with each. Baseline before the audit:
`go test -count=1 -p 1 ./...` and `go test -count=1 -p 1 -tags oracle ./...` both
pass.

Summary: the engine core is in genuinely good shape — a dozen adversarial
differential probes (pathological package names, multiple stow directories,
adopt conflicts in both directions, compat+dotfiles unstow, stale-link restow,
`.stowrc` containing `--help`/`-V`/bad options) all matched the oracle exactly.
The bugs that remain live where the suite has no fixtures: **error paths and
non-zero verbosity on non-happy paths**.

---

## Bugs

Behaviour differs from real stow 2.4.1, or from gostow's own documented intent.
Ranked by severity.

### B1 — The level-2 `CONFLICT when stowing ...` line is never printed  **[confirmed]**

- **Where:** `stow/engine.go:252` (`engine.conflict`); referent `Stow.pm:1389`.
- **What:** `Stow.pm`'s `conflict()` emits
  `debug(2, 0, "CONFLICT when ${action}ing $package: $message")` the moment a
  conflict is recorded. gostow's `conflict()` only appends to the slice. At
  `-vv`, real stow prints one `CONFLICT when stowing pkg: <message>` line per
  conflict, interleaved with the planning trace; gostow prints none of them.
- **Why it's wrong:** verbosity 2 is inside the byte-exact contract (SPEC §8.2,
  PL-11 ruling). This is a scriptable line gostow silently drops.
- **Demonstrated:** package file over an existing plain target file,
  `stow -d stow -t target -vv pkg` — oracle stderr contains the CONFLICT line
  between `level of f is 0` and `Planning stow of package pkg... done`; gostow's
  stderr is identical except that line is absent. Exit codes agree (1).
- **Why the suite missed it:** every conflict fixture runs at verbosity 0, and
  `TestVerbosityZeroToTwoIsASubsequenceOfHigherLevels` compares gostow to
  *itself*, so a line missing at every level is invisible to it. See H2.

### B2 — Fatal error messages embed Go error text where stow embeds `strerror(errno)`  **[confirmed]**

- **Where:** every `fatalf` that formats an error with `%v`:
  `stow/engine.go:308` (`readdirSorted`), `stow/engine.go:410`,
  `stow/tasks.go:210` (`readALink`), `stow/tasks.go:261` (`doUnlink`),
  `stow/tasks.go:332-349` (`processTask`, five messages).
- **What:** stow's messages end `($!)` — glibc's `strerror`, e.g.
  `(Permission denied)`. gostow substitutes Go's error string, which for a
  `*PathError` includes the syscall name and the **absolute** path.
- **Demonstrated:** package with a `chmod 000` subdirectory:
  - oracle: `stow: ERROR: cannot read directory: ../stow/pkg/sub (Permission denied)`
  - gostow: `gostow: ERROR: cannot read directory: ../stow/pkg/sub (open /tmp/.../stow/pkg/sub: permission denied)`
- **Why it's wrong:** SPEC §8.3 states "both message forms are replicated
  verbatim". The errno *string* is reproducible and dependable (tier 1) — only
  the exit status is ruled tier-2 (PL-07). The Go text also leaks an absolute
  path into output that otherwise speaks target-relative paths, undoing the
  care `engine.real()` takes. A fix would map `errors.As(&fs.PathError)` /
  `syscall.Errno` to `strerror`-style text (`unix.ErrnoName` is not it —
  `Errno.Error()` already returns the strerror string, capitalised like Perl's).
- **Severity note:** these paths fire on real systems (permissions, EXDEV on
  `--adopt` across filesystems, ENOSPC), and none of them has a differential
  fixture.

### B3 — A directory named `.stowrc` is silently ignored; real stow dies and stows nothing  **[confirmed]**

- **Where:** `internal/cli/stowrc.go:40-52` — the `bufio.Scanner` loop never
  checks `sc.Err()`.
- **What:** on Linux, `open(2)` of a directory with `O_RDONLY` succeeds; the
  first read fails `EISDIR`. Perl's failed readline poisons the handle so
  `close` fails, and stow dies: `Could not close open file: .stowrc`, exit 21
  (errno-derived; gostow's pin would be 2), **nothing stowed**. gostow's
  `f.Close()` succeeds (Go doesn't propagate read errors to Close), the read
  error sits unexamined in `sc.Err()`, and gostow proceeds as if the rc file
  were empty: exit 0, package stowed.
- **Why it's wrong:** stderr bytes, exit status *and the resulting tree* all
  diverge. It is also the same bug shape as PL-10's gostow episode ("`os.Open`
  succeeds on a directory and the reader swallowed the EISDIR") — fixed in
  `parseIgnoreReader`, still present in `readStowrcTokens`.
- **Demonstrated:** `mkdir sandbox/.stowrc`, then `stow -d stow -t target pkg`
  from the sandbox root: oracle exit 21 + die message + empty target; gostow
  exit 0, silent, `target/f` created.
- **Also implied (unconfirmed):** any mid-file read error, and any `.stowrc`
  line over `bufio.Scanner`'s 64 KiB default limit, is silently treated as
  end-of-file — a truncated config, honoured without a word.

### B4 — A target (or stow) directory that exists but cannot be entered: stow dies in `canon_path`, gostow reports success  **[confirmed]**

- **Where:** `stow/engine.go:240-250` — `canonPath`'s
  `//nolint:nilerr` fallback returns the absolute path on *any*
  `EvalSymlinks` failure, not only ENOENT.
- **What:** `Stow::Util::canon_path` canonicalises by `chdir` and dies when it
  cannot: `stow: ERROR: canon_path: cannot chdir to target from <cwd>`, exit 13
  (EACCES). gostow swallows the failure and plans against paths it cannot read.
- **Demonstrated:** `chmod 644 target` (no search bit), then
  `stow -d stow -t target -n pkg`: oracle exits 13 with the ERROR line; gostow
  prints only the simulation warning and exits **0**. In non-simulate mode
  gostow would instead fail later with a `Could not create symlink` message
  (which then hits B2's wording problem).
- **Why it's wrong:** simulate mode is exactly the mode people use to check
  whether a run would work; answering "yes" where stow answers "no, fatally"
  is a behavioural divergence, not just bytes. The comment on `canonPath`
  claims the fallback exists for "a not-yet-existing path" — the nolint
  swallows much more than that.

### B5 — Invalid `--ignore`/`--defer`/`--override` regex: wrong timing, wrong exit code, no usage block  **[confirmed]**

- **Where:** compilation happens in `stow/engine.go:213-221` (`newEngine`),
  i.e. after `.stowrc` handling, `--dir`/`--target` validation and the
  "No packages" check. Real stow compiles `qr{($regex)\z}` inside the
  Getopt::Long *callback* (`bin/stow:582-597`), so a bad pattern is a **parse
  failure**: diagnostic on stderr, full usage block on stdout, **exit 1** —
  deterministically, not errno-derived.
- **Demonstrated:** `stow -d stow -t target --ignore='(' pkg`:
  - oracle: Perl regex diagnostic on stderr, usage on stdout, exit **1**
  - gostow: `gostow: ERROR: invalid regex "(": ...`, no usage block, exit **2**
  and with *no* packages, the oracle still reports the regex error while gostow
  reports `No packages to stow or unstow` — the bad pattern is never noticed.
- **Why it's wrong:** the message *text* is Perl-engine output and cannot be
  replicated (PL-19's situation), but the **shape** — parse-time detection,
  usage-on-stdout, exit 1 — is reproducible tier-1 behaviour, and the current
  divergence is recorded nowhere. This needs either a fix (validate at parse
  time, emit usage, exit 1) or a ledger entry ruling the substitute.

### B6 — Unknown user in `~user` expansion: stow substitutes the empty string, gostow leaves the token  **[confirmed]**

- **Where:** `internal/cli/stowrc.go:197-199` — `user.Lookup` failure returns
  the match unchanged; Perl's `(getpwnam($1))[7]` is undef, which interpolates
  as `""` (plus a `Use of uninitialized value` warning on stderr).
- **Demonstrated:** `.stowrc` containing `--target=~nosuchuser99/x`:
  - oracle: warning line, then `stow: --target value '/x' is not a valid directory`, exit 1
  - gostow: `gostow: --target value '~nosuchuser99/x' is not a valid directory`, exit 1
  And the sharper case `--target=~nosuchuser99<real-abs-path>`: the oracle
  *collapses the tilde and stows into the real path, exit 0*; gostow refuses,
  exit 1. Tree, exit code and stderr all diverge.
- **Ruling needed:** the empty-string substitution is Perl-falsiness-adjacent
  but deterministic; the uninitialised-value warning bytes are Perl-internal
  (tier 2). Either replicate the substitution (and rule the warning
  unreproducible) or ledger the divergence. Today it is neither ruled nor
  recorded — `docs/DIVERGENCES.md` claims the difference list is exhaustive.

## Hazards

Tests that cannot fail, and duplication waiting to diverge.

### H1 — TEST-PLAN's L5 goldens layer does not exist

`docs/TEST-PLAN.md` §2 and §3 describe goldens "generated by L4, frozen in
`testdata/`", an `-update-goldens` flag gated on `stow --version` = 2.4.1, and
claim "`go test ./...` runs L1, L2, L3, **L5**" and "goldens make the everyday
suite hermetic; L4 is what keeps the goldens honest". None of this is in the
tree: no golden files, no flag (`grep -r update-goldens` finds only the doc).
Measured consequence (hermetic run, `-coverpkg=./...`): `doMv` (--adopt's move)
0%, `isRealDir` 0%, `expandTilde` 14%, `sanitizePaths` 39% — the adopt move,
the no-folding real-dir test, tilde expansion and default dir/target resolution
execute **only** when Perl and the pinned stow are installed. CI does run the
oracle job on every PR, so the gate holds there, but a contributor's local
`go test ./...` is materially weaker than the document promises. Either build
L5 or rewrite TEST-PLAN to say the truth.

### H2 — No differential fixture runs a conflict, an unstow, or a fatal error above verbosity 0

The `-v`/`-vv` fixtures are all clean stows of one file. Conflicts, unstows,
unfolds and fatal errors are exercised at verbosity 0 only (engine oracle cases
run at `-v 1`). That is exactly the hole B1 lived in, and it means every other
level-2 line on a non-happy path (`CONFLICT ...`, `Cleaning up any invalid
links ...` on real fixtures, `--- removing invalid link ...`) is pinned by at
most one code path. The subsequence test can never catch an everywhere-missing
line because it compares gostow to itself. Cheap fix with existing machinery:
run the conflict, unstow-with-cleanup, and unfold fixtures again at `-vv`.

### H3 — `Apply` silently ignores an unknown `Request.Action`

`stow/engine.go:147-157`: the `switch` has no default, so
`Apply(opts, Request{Action: 7, Packages: ...})` plans nothing and returns
success. For a semver-bound library that dstow will drive programmatically, a
silent no-op on a caller bug is the worst available behaviour; an error (or
even a panic — it is a programmer error, like a bad `TaskAction` which *does*
get `fatalf("bad task action")`) would be consistent with the engine's own
treatment of impossible states.

### H4 — `joinPaths`/`canonpath`/`removeParentRefs` are transcriptions with hand-written expectations

The `parent` bug was born as a transcription verified only by hand-written
cases, and the fix added a live Perl oracle (39 paths against
`Stow::Util::parent`). `joinPaths` and its two helpers are the same shape of
risk — ported Perl, pinned by a hand-port of `join_paths.t` (22 cases) — and
have no oracle sweep. I verified the port line-by-line against `Util.pm` in
this audit (including the `length $part` guard, so `"0"` fragments survive —
no Perl-falsiness slip), so I found no live bug; the hazard is structural. The
`parentOracle` harness in `internal/stowpath/parent_oracle_test.go:49` would
take an hour to generalise to `join_paths` over the same adversarial path list.

## Smells

### S1 — `cleanupInvalidLinks`' warning drops the action word

`stow/engine.go:719`: `"Unexpected action scheduled for %s; skipping clean-up"`.
`Stow.pm:1185`: `"Unexpected action $action scheduled for $node_path; skipping clean-up"`.
Both branches are unreachable today (cleanup runs only during the unstow phase,
which wholly precedes any phase that files a create task against an on-disk
symlink — same argument as PL-06), so this cannot diverge observably. But it is
a transcription drift inside code whose whole justification is byte parity; if
the phase structure ever changes, it diverges silently. Either match the bytes
or comment why not.

### S2 — One `readdirSorted` wording where stow has two

stow says `cannot read directory: $dir ($!)` in `stow_contents`,
`unstow_contents` and `cleanup_invalid_links`, but
`Cannot read directory "$dir" ($!)\n` — capital C, quoted path, embedded
newline — in `foldable` and `fold_tree` (`Stow.pm:1256,1350`). gostow's single
printer (`stow/engine.go:307-318`) always uses the first form. Reachability
analysis says the `foldable`/`fold_tree` opens are always preceded by an open
of the same directory that would fail first with the other message, so only a
mid-run race can expose it. Worth a one-line comment on `readdirSorted` citing
that argument, so nobody "fixes" it in either direction blind.

### S3 — `canonpath` vs `canonPath` (the question the kickoff asked)

Verdict: **defensible, marginally**. They port two different upstream routines
whose names are equally confusable in Perl (`File::Spec->canonpath` vs
`Stow::Util::canon_path`), both carry doc comments saying so, and their
signatures differ (`string → string` vs `string → (string, error)`), so most
misuses fail to compile. What tips it to a smell anyway: the distinction is
case-only, invisible in speech and in review diffs, and the functions live in
different files of the same package. A rename that keeps the upstream lineage
visible — e.g. `perlCanonpath` for the `File::Spec` port — costs one commit and
removes the trap. Do it or don't, but decide once; do not let a future reader
"unify" them, which is the real failure mode (they are *not* the same
operation — `canonpath` must leave `a/../b` alone).

### S4 — Dead and untested declared API

- `Action.String()` (`stow/engine.go:27`): no callers. It exists so `%v` prints
  something sane and is documented as diagnostics-only; fine to keep, but it is
  untested — a one-line assertion would stop a rename from silently changing
  `%v` output in future library-consumer logs.
- `stow.Restow` (`stow/engine.go:132`): 0% coverage. `ActionRestow` itself is
  oracle-tested via `Apply`; only the three-line sugar is unexercised. A
  library consumer's first call shouldn't be the first execution ever.
- `gerund`'s `restowing` branch (`internal/cli/cli.go:266`): unreachable — the
  engine files every conflict as `ActionStow` (as does stow 2.4.1; grep of
  `conflict(` in `Stow.pm` shows only `'stow'`). Harmless, mirrors upstream's
  own dead `'unstow'` reporting loop, but worth a comment.
- `ConflictError.Error` / `FatalError.Error` / `dieError.Error`: 0% only in the
  sense that nothing asserts their text; they run constantly via `errors.As`
  machinery. Not worth action.

### S5 — Stale comments

- `internal/conformance/oracle_smoke_test.go:8`: "gostow has no engine yet" —
  the engine has been done for a day and the comment already reads like a fossil.
- `stow/paths_test.go:46`: "Ported from stow 2.4.1's t/parent.t (5 assertions)."
  dangles above nothing; the parent tests moved to `internal/stowpath`.
- `internal/stowpath/path.go:22` duplicates `reSlashRun` from `stow/paths.go:18`
  — two compiles of `/+`, trivial, but the kind of copy the kickoff asks about.

## Suspected but unconfirmed

What I would have run, and what would decide it.

### U1 — Perl-valid, RE2-invalid patterns in `--ignore`/`--defer`/`--override` and ignore files

`stow --ignore='foo(?!bar)' pkg` works in real stow and is a fatal error in
gostow (RE2 has no lookaround). SPEC §11.5 rules "RE2 first, escape hatch
later" for ignore *files*; nothing extends that ruling to the three CLI flags,
and B5 shows their error path already diverges. To confirm:
`stow -d stow -t target --ignore='x(?!y)' pkg` against both binaries — oracle
stows with the pattern honoured, gostow exits 2. Decide: extend the §11.5
ruling to flags explicitly (ledger entry), or vendor the fallback engine.

### U2 — `.stow-local-ignore` with a line over 64 KiB

`parseIgnoreReader` uses a default `bufio.Scanner` (`stow/ignore.go:184`),
which errors at 64 KiB per line; stow has no such limit. Unlike B3 this error
*is* propagated, so gostow would fail loudly where stow works. Confirm:
generate a 100 KB single-line ignore file, run both. Fix is
`sc.Buffer(nil, 1<<20)` or a `bufio.Reader` loop.

### U3 — Getopt::Long callback `die` semantics beyond regexes

B5 shows a callback `die` becomes "print diagnostic, GetOptions returns false,
usage('')". The 6307-vector fuzz corpus cannot see this because the vectors
exercise the *parser*, not `bin/stow`'s callbacks. Worth one probe each:
`--ignore=` (empty value — stow compiles `qr{()\z}` fine; gostow?) and repeated
`--ignore` flags mixing valid and invalid patterns (which diagnostics
accumulate?). The empty-pattern case:
`stow -d stow -t target --ignore= pkg` — Getopt treats `--ignore=` as a missing
argument (`Option ignore requires an argument`)? The vectors likely cover the
parse; what they do not cover is that gostow then *compiles* `(``)$` — matches
everything as a suffix? — versus stow's `qr{()\z}` which matches every string
too. Probably parity by accident; one fixture would pin it.

### U4 — `--adopt` across filesystem boundaries

`os.Rename` fails EXDEV where Perl's `rename` also fails EXDEV — parity likely
holds on the *event*, but the message diverges per B2 and neither suite has a
fixture (needs two mount points; a tmpfs in CI would do). Confirm alongside B2's
fix.

### U5 — The Perl uninitialised-value warning bytes in B6's scenario

If B6 is ruled "replicate the empty-string expansion", the accompanying
`Use of uninitialized value in substitution iterator at .../stow line 775.`
line is still Perl-internal, references the oracle's install path, and varies
with Perl version — unreplicable, needs a tier-2 ledger entry exactly like
PL-19's.

## Probes that came back clean (evidence, not findings)

All matched the oracle byte-for-byte (stderr/stdout normalised for sandbox
root), same exit, same tree, this session:

- `stow ///` (empty package name after slash-strip), `stow .`, `stow ..`
- unfold pulling a second package from a *different*, `.stow`-marked stow dir
- `--adopt` conflict both directions (dir-over-file, file-over-dir)
- `--dotfiles --compat -D` (reverse translation on the target walk)
- restow with a renamed package file (stale-link cleanup on the real binary)
- `--ignore` matching the *untranslated* `dot-` name under `--dotfiles`
- `.stowrc` containing `--help`, `-V`, and an unknown option (identity-line
  divergence only, which is ruled)
- unstow of a never-stowed package; target inside the stow dir (both refuse)

Mutation checks run and passed (guard went red, restored, green):
`TestHelpDocumentsEveryOption` (new flag in `spec()` without help text) and
`TestFixQuirksGivesRmdirItsColon` (`opRmdir.colon = true`).

## Settled decisions reviewed

No grounds found to reopen any: the help-prose ruling, MIT+NOTICE, `chkstow`
exclusion, and the ledger rulings all held up against the source and the
binary. The one settled-adjacent item this audit asks a human to decide is B5/B6:
both need either a fix or a ledger entry, because today `docs/DIVERGENCES.md`'s
claim to list every difference exhaustively is false in those two corners.
